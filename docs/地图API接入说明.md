# 地图 API 接入说明

当前项目已经进入真实地图接入阶段：页面主体开始从 Mock 地图容器切换到百度地图，景点和城市仍以自有后端数据为主。

## 为什么先 Mock 再接真实地图

- 地图 API 涉及 Key、域名、安全配置和额度限制。
- 先 Mock 可以专注完成页面结构、接口契约和数据模型。
- 后续替换为真实地图时，只需要替换地图组件内部实现。

## 当前配置要求

1. 注册或登录百度地图开放平台。
2. 创建浏览器端应用。
3. 申请 JavaScript API 使用的 AK。
4. 配置安全域名：
   - 本地开发通常需要允许 `localhost` 或 `127.0.0.1`。
   - 正式部署后需要配置真实线上域名。
5. 如果后续使用服务端能力，还需要额外创建服务端应用并确认配额。

## 前端环境变量规划

前端后续使用 `.env.local` 保存本地 Key：

```plain
VITE_BAIDU_MAP_AK=你的百度地图浏览器端AK
```

真实 AK 不应写死在源码里，也不应提交到仓库。

## 后端环境变量规划

如果后端调用百度地图服务端能力，后续使用环境变量或配置文件占位：

```plain
BAIDU_MAP_SERVER_AK=你的百度地图服务端AK
```

后端负责路线规划时，接口可以先设计为：

```plain
POST /api/routes/plan
```

请求内容包含城市、起点、景点列表、交通方式和规划模式。返回内容不是单纯路线，而是“景点顺序 + 路线分段 + 停留时间 + 总时间”的行程结果。

当前先约定一版行程规划契约，后续允许迭代：

- 请求核心字段：
  - `cityId`
  - `startPoint`
  - `endPoint`
  - `spotIds`
  - `transportType`
  - `planMode`
- 完整行程模式预留字段：
  - `travelDate`
  - `tripDays`
  - `dailyStartTime`
  - `dailyEndTime`
  - `includeLunchBreak`
  - `includeNightTour`
  - `intensity`
- 响应核心字段：
  - `orderedSpotIds`
  - `segments`
  - `spotStayPlans`
  - `totalTravelDurationSeconds`
  - `totalStayDurationMinutes`
  - `totalTripDurationMinutes`
  - `itineraryDays`

时间口径约定：

- `totalTravelDurationSeconds`：纯交通耗时总和
- `totalStayDurationMinutes`：景点建议停留时长总和
- `totalTripDurationMinutes`：交通耗时 + 停留时长，按分钟汇总后的总行程时间
- `spotStayPlans`：自由路线模式先返回每个景点建议游玩时长；完整行程模式再补建议到达/离开时间

当前实现边界：

- 已落地后端接口：`POST /api/routes/plan`
- 前端行程池已接通该接口，并会在地图上绘制返回的路线线条
- 右侧路线抽屉的分段颜色与地图路线颜色保持一致，方便用户对应查看每一段
- 路线规划成功后，右侧改为展示路线详情抽屉，景点详情面板临时关闭
- 底部行程池改成单行工具条：左侧为景点池入口，右侧为规划参数、核心指标和开始规划按钮
- 当前优先支持 `free` 自由路线模式
- `schedule` 完整行程模式第一版已接入“多天拆分 + 时间表生成”，当前按默认作息生成 `itineraryDays`，酒店推荐和餐厅推荐暂不作为第一版范围
- `schedule` 完整行程模式已补充午餐、休息、酒店节点：
  - 支持“系统推荐 / 手动输入 / 不安排”
  - 手动输入地点会在前端提交前解析坐标
  - 系统推荐会按当前景点附近实时检索 POI；若未命中或未配置 AK，则退回为普通时间块
  - 对于有坐标的补充节点，后端会按当天完整节点顺序重算真实路线段，不再只用前端虚直线引导
- 完整行程模式点击“开始规划”后会先弹出详细配置弹窗；生成后页面切换为“地图上方 Day 标签 + 右侧当天详细行程 + 悬浮设置按钮”的结果布局
- 行程设置里的酒店、午餐、休息和返回酒店，当前都已开始参与完整行程编排；偏好标签仍先保留为前端展示预留
- 完整行程右侧步骤条和地图路线颜色当前已统一按“当天实际路线段顺序”渲染，避免插入午餐或酒店节点后颜色错位
- 当前交通方式先只保留 `公共交通 / 驾车 / 步行 / 骑行`，暂不提供单独的 `打车`
- 景点顺序暂按景点池传入顺序执行，尚未接入“一键智能排序”
- 有服务端 AK 时调用百度轻量级路线规划服务；未配置 AK 或第三方失败时，后端会回退为本地估算路线，保证主流程可调试
- 起点输入框支持地点联想候选，选中后直接绑定坐标
- 起点联想请求会做前端防抖，并优先按当前城市做地点候选检索
- 完整行程里的酒店、午餐、休息“用户自定义地点”已补为联想输入：
  - 输入 2 个字后触发 POI 候选检索
  - 前端会优先按已选景点中心点对候选做近距离排序
  - 同名连锁地点仍建议继续输入商圈、景区或道路名缩小范围
- 起点支持“当前位置”快捷填充，浏览器定位坐标会先转为 GCJ-02 再提交
- 起点为必填项；若未显式选中候选项，则在点击规划时再尝试一次地点检索，仍未命中时提示用户重新选择，不再回退到城市中心点
- 路线详情中的每个交通分段支持通过官方 URI 跳转到百度或高德地图网站查看实时详细路线
  - 百度使用 `api.map.baidu.com/direction`，直接传项目 GCJ-02 坐标并声明 `coord_type=gcj02`
  - 高德使用 `uri.amap.com/navigation`，项目 GCJ-02 坐标可直接使用，禁止额外转换为 BD-09
  - 不依赖 `map.baidu.com/dir`、`ditu.amap.com/ssr/dir` 等地图网站内部分享参数
  - 百度网页路线入口暂不提供骑行模式，高德骑行路线优先在移动端查看
- 完整行程模式当前会按默认参数生成多天结果：默认 2 天、09:00-18:00、标准强度；前端详细参数输入后续再补

## 当前接入范围

1. 前端加载百度地图 JavaScript API。
2. 根据城市中心点定位地图。
3. 根据景点经纬度绘制 Marker。
4. 点击 Marker 后联动右侧景点详情。
5. 对已配置轮廓的区域型景点，选中后优先展示 Polygon 轮廓并自动缩放到范围。
6. 地图默认保持城市全景，选中景点后使用中等缩放聚焦具体位置，便于看清景点落点。
7. 工作台默认不选中任何景点；筛选、搜索只更新点位和列表，不自动重置为首个景点，地图视角只在切换城市或主动选点时变化。
8. 顶部筛选、搜索和推荐排序切换时不重新初始化地图实例，只保留当前底图并更新 Marker 数据。

## 精准点位接入

当前后端已补充 POI 校准候选接口，用来给精选景点查更准确的主点位：

```plain
GET /api/poi-calibration/candidates?cityName=西安市&keyword=大唐不夜城&addressKeyword=慈恩路
```

用途说明：

- 这个接口不是给普通用户搜索景点，而是给开发/运营阶段校准精选景点用。
- 它会调用百度地点检索，返回最多 5 个候选 POI，便于人工挑选更准确的主点位。
- 适合先修正景点主点位，再决定是否补轮廓范围。

当前仓库已经先按这条思路落了一版西安精选景点校准：

- 单体景点优先修正主点位
- 区域型景点在轮廓不准时先回退为主点位展示
- 以独立 Flyway 迁移版本保存，便于回溯和继续迭代

当前精选景点已继续按百度地点检索主 POI 结果完成一轮全量坐标校准，现阶段统一采用“景点主点位”展示策略，不再展示不可靠轮廓。

## 管理端地图资料维护

城市和景点新增表单已接入百度地图服务，服务端 AK 始终由后端持有：

```plain
GET /api/admin/map-data/provinces
GET /api/admin/map-data/cities?provinceCode=320000
GET /api/admin/map-data/city-location?provinceCode=320000&cityCode=320100
GET /api/admin/map-data/spot-candidates?cityName=南京市&keyword=南京博物院
```

- 以上接口和其他 `/api/admin/**` 接口一样仅允许 `ADMIN` 角色访问。
- 城市编码统一使用六位行政区划 `adcode`，不是拼音编码；省市关系由后端再次校验。
- 城市中心点和景点主点位统一返回 GCJ-02，写入数据库前保留六位小数。
- 景点联想需要先选择所属城市，并由管理员确认具体候选项，不自动采用第一条结果。
- 百度候选不准确时，可以在管理端百度地图上点击选点；浏览器返回的 BD-09 会转换成 GCJ-02 后提交。
- 本期不使用百度 UID 或高德 POI ID 作为业务关联字段，平台 POI ID 不参与地图展示和路线规划。
- `boundary_geojson` 仍是可选高级字段，当前只支持 GCJ-02 `Polygon` 的第一条闭合外环，不支持运行时自动获取 AOI 边界。

## 下一步规划

5. 在完整行程模式中补充“吃饭 / 休息 / 住宿”这类实时地点能力，优先采用运行时地点检索，不做全量自建库。
6. 允许用户手动选择酒店、商场、餐饮等中间停靠点，并支持系统按景点邻近性推荐候选 POI。
7. 在时间编排结果中把午餐、晚餐、午休、回酒店等时间块和真实地点绑定起来，而不只是展示抽象占位块。
8. 精选景点批量校准/导入工具暂不在当前阶段开发，等主流程稳定后再补批量处理能力。

## 风险提醒

- 地图 Key 泄露会带来额度消耗风险。
- 前端 Key 需要配置安全域名，后端 Key 不要暴露给浏览器。
- 路线规划结果依赖第三方 API，失败时需要有降级提示。
- 多端扩展时，App 和小程序可能需要分别申请不同类型的 Key。
- 浏览器端 AK 用于前端地图展示；服务端 AK 只用于后端调用，不应混用。
- 百度地图 JavaScript API 默认使用 BD09 坐标系。当前前端已在百度地图适配层中把后端返回的 GCJ-02 坐标转换为 BD09，用于修正点位偏移。
- 如果后续后端直接调用百度地图服务端能力，也要在服务端保持同样的坐标适配策略，避免前后端结果不一致。
- 景点轮廓不应依赖运行时搜索结果临时生成，精选景点更适合在后端提前沉淀主点位和可选边界数据。
