# 地图 API 接入说明

当前项目已经进入真实地图接入阶段：页面主体开始从 Mock 地图容器切换到百度地图，景点和城市仍以自有后端数据为主。

## 为什么先 Mock 再接真实地图

- 地图 API 涉及 Key、域名、安全配置和额度限制。
- 先 Mock 可以专注完成页面结构、接口契约和数据模型。
- 后续替换为真实地图时，只需要替换地图组件内部实现。

## 当前配置要求

1. 注册或登录百度地图开放平台。
2. 创建浏览器端应用。
3. 申请 JavaScript API 使用的 AK。
4. 配置安全域名：
   - 本地开发通常需要允许 `localhost` 或 `127.0.0.1`。
   - 正式部署后需要配置真实线上域名。
5. 如果后续使用服务端能力，还需要额外创建服务端应用并确认配额。

## 前端环境变量规划

前端后续使用 `.env.local` 保存本地 Key：

```plain
VITE_BAIDU_MAP_AK=你的百度地图浏览器端AK
```

真实 AK 不应写死在源码里，也不应提交到仓库。

## 后端环境变量规划

如果后端调用百度地图服务端能力，后续使用环境变量或配置文件占位：

```plain
BAIDU_MAP_SERVER_AK=你的百度地图服务端AK
```

后端负责路线规划时，接口可以先设计为：

```plain
POST /api/routes/plan
```

请求内容包含城市、起点、景点列表、交通方式和规划模式。返回内容不是单纯路线，而是“景点顺序 + 路线分段 + 停留时间 + 总时间”的行程结果。

当前先约定一版行程规划契约，后续允许迭代：

- 请求核心字段：
  - `cityId`
  - `startPoint`
  - `endPoint`
  - `spotIds`
  - `transportType`
  - `planMode`
- 完整行程模式预留字段：
  - `travelDate`
  - `tripDays`
  - `dailyStartTime`
  - `dailyEndTime`
  - `includeLunchBreak`
  - `includeNightTour`
  - `intensity`
- 响应核心字段：
  - `orderedSpotIds`
  - `segments`
  - `spotStayPlans`
  - `totalTravelDurationSeconds`
  - `totalStayDurationMinutes`
  - `totalTripDurationMinutes`
  - `itineraryDays`

时间口径约定：

- `totalTravelDurationSeconds`：纯交通耗时总和
- `totalStayDurationMinutes`：景点建议停留时长总和
- `totalTripDurationMinutes`：交通耗时 + 停留时长，按分钟汇总后的总行程时间
- `spotStayPlans`：自由路线模式先返回每个景点建议游玩时长；完整行程模式再补建议到达/离开时间

当前实现边界：

- 已落地后端接口：`POST /api/routes/plan`
- 前端行程池已接通该接口，并会在地图上绘制返回的路线线条
- 右侧路线抽屉的分段颜色与地图路线颜色保持一致，方便用户对应查看每一段
- 路线规划成功后，右侧改为展示路线详情抽屉，景点详情面板临时关闭
- 底部行程池改成单行工具条：左侧为景点池入口，右侧为规划参数、核心指标和开始规划按钮
- 当前优先支持 `free` 自由路线模式
- 当前交通方式先只保留 `公共交通 / 驾车 / 步行 / 骑行`，暂不提供单独的 `打车`
- 景点顺序暂按景点池传入顺序执行，尚未接入“一键智能排序”
- 有服务端 AK 时调用百度轻量级路线规划服务；未配置 AK 或第三方失败时，后端会回退为本地估算路线，保证主流程可调试
- 起点输入框支持地点联想候选，选中后直接绑定坐标
- 起点联想请求会做前端防抖，并优先按当前城市做地点候选检索
- 起点支持“当前位置”快捷填充，浏览器定位坐标会先转为 GCJ-02 再提交
- 若起点未显式选中候选项，则在点击规划时再尝试一次地点检索；仍失败才回退到所选城市中心点

## 当前接入范围

1. 前端加载百度地图 JavaScript API。
2. 根据城市中心点定位地图。
3. 根据景点经纬度绘制 Marker。
4. 点击 Marker 后联动右侧景点详情。
5. 对已配置轮廓的区域型景点，选中后优先展示 Polygon 轮廓并自动缩放到范围。
6. 地图默认保持城市全景，选中景点后使用中等缩放聚焦具体位置，便于看清景点落点。
7. 工作台默认不选中任何景点；筛选、搜索只更新点位和列表，不自动重置为首个景点，地图视角只在切换城市或主动选点时变化。
8. 顶部筛选、搜索和推荐排序切换时不重新初始化地图实例，只保留当前底图并更新 Marker 数据。

## 精准点位接入

当前后端已补充 POI 校准候选接口，用来给精选景点查更准确的主点位：

```plain
GET /api/poi-calibration/candidates?cityName=西安市&keyword=大唐不夜城&addressKeyword=慈恩路
```

用途说明：

- 这个接口不是给普通用户搜索景点，而是给开发/运营阶段校准精选景点用。
- 它会调用百度地点检索，返回最多 5 个候选 POI，便于人工挑选更准确的主点位。
- 适合先修正景点主点位，再决定是否补轮廓范围。

当前仓库已经先按这条思路落了一版西安精选景点校准：

- 单体景点优先修正主点位
- 区域型景点在轮廓不准时先回退为主点位展示
- 以独立 Flyway 迁移版本保存，便于回溯和继续迭代

当前精选景点已继续按百度地点检索主 POI 结果完成一轮全量坐标校准，现阶段统一采用“景点主点位”展示策略，不再展示不可靠轮廓。

## 风险提醒

- 地图 Key 泄露会带来额度消耗风险。
- 前端 Key 需要配置安全域名，后端 Key 不要暴露给浏览器。
- 路线规划结果依赖第三方 API，失败时需要有降级提示。
- 多端扩展时，App 和小程序可能需要分别申请不同类型的 Key。
- 浏览器端 AK 用于前端地图展示；服务端 AK 只用于后端调用，不应混用。
- 百度地图 JavaScript API 默认使用 BD09 坐标系。当前前端已在百度地图适配层中把后端返回的 GCJ-02 坐标转换为 BD09，用于修正点位偏移。
- 如果后续后端直接调用百度地图服务端能力，也要在服务端保持同样的坐标适配策略，避免前后端结果不一致。
- 景点轮廓不应依赖运行时搜索结果临时生成，精选景点更适合在后端提前沉淀主点位和可选边界数据。
