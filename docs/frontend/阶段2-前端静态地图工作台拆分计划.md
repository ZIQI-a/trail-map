# 阶段 2：前端静态地图工作台拆分计划

本文档用于把“前端静态地图工作台”拆成多个可审查的小步骤。每次只开发一个小模块，完成后你审查通过，再进入下一小步。

## 阶段目标

按照 `require/原型图.png` 搭建地图工作台静态页面。此阶段只使用前端 Mock 数据，不接后端接口，不接真实百度地图 API。

当前状态：已完成。

## 开发原则

- 一次只做一个小模块，避免页面、状态、地图、接口混在一起。
- 每个小模块都要能单独运行、单独审查。
- 每次改动后执行 `npm run lint` 和 `npm run build`。
- 默认不启动本地开发服务；页面预览由用户本地运行，AI 只做质量验证。
- 页面从一开始考虑 H5 兼容，小屏不允许横向溢出。
- 先实现结构和 Mock 数据，再逐步补交互，不提前接真实地图 API。
- 基础 UI 控件使用 Ant Design，业务布局和地图浮层使用 CSS Modules。

## 前端结构约定

- `src/pages`：路由级页面入口。页面只负责布局编排和跨组件共享状态，例如当前选中景点。
- `src/components`：页面组件和可复用组件目录。地图工作台组件统一放在 `src/components/map-workbench`。
- `src/mocks`：前端 Mock 数据目录。地图工作台 Mock 数据统一放在 `src/mocks/map-workbench`。
- `src/utils`：工具函数目录。地图工作台工具函数统一放在 `src/utils/map-workbench`。
- `src/types`：跨模块共享类型。阶段 2 的 Mock 数据类型先放这里，方便后续对齐后端接口。
- 后续小模块必须优先新增独立组件，不把新功能继续堆进 `MapWorkbenchPage.tsx`。

## 小模块 2.1：前端目录与基础布局骨架

### 目标

建立后续页面开发需要的目录结构，并把当前默认页替换为“地图工作台”的基础外壳。

### 开发范围

- 新建推荐目录：`src/pages`、`src/components/map-workbench`、`src/mocks/map-workbench`、`src/utils/map-workbench`、`src/types`。
- 创建地图工作台页面入口。
- 页面先只包含基础区域：顶部栏、主内容区、底部规划区占位。
- 不写景点列表、详情卡片、地图点位等具体业务内容。

### 验收标准

- 页面能看出整体工作台结构。
- 桌面端和移动端都不横向溢出。
- `npm run lint` 和 `npm run build` 通过。

### 教学重点

理解为什么先拆页面骨架：骨架稳定后，后续模块只是在固定区域内补内容，复杂度更低。

## 小模块 2.2：Mock 数据与类型定义

### 目标

定义前端临时数据结构，为后续景点列表、详情、筛选和行程池提供统一数据来源。

### 开发范围

- 新建 `src/types` 中的城市、景点、标签、交通方式等 TypeScript 类型。
- 新建 `src/mocks/map-workbench` 保存成都示例数据。
- 数据字段先贴近数据库草案，但只保留页面需要展示的字段。

### 验收标准

- Mock 数据可以被页面导入使用。
- 类型命名清晰，后续可以平滑替换为后端接口类型。
- 不出现真实地图 Key 或真实接口地址。

### 教学重点

理解 TypeScript 类型的作用：它是前后端接口联调前的“数据契约草稿”。

## 小模块 2.3：顶部导航与分类筛选

### 目标

实现原型顶部区域：品牌、搜索框、城市选择、操作按钮、景点分类筛选。

### 开发范围

- 顶部品牌和搜索输入框。
- 当前城市展示，先固定为“成都市”。
- 我的定位、路线规划、收藏夹等按钮先做静态按钮。
- 分类筛选只做前端选中状态，不接真实筛选逻辑。

### 验收标准

- 顶部区域视觉接近原型。
- 分类按钮有清晰的选中态。
- 小屏下顶部内容能换行或折叠，不挤出屏幕。

### 教学重点

理解“静态 UI”和“真实业务逻辑”的边界：先让界面和状态跑起来，再接数据。

## 小模块 2.4：Mock 地图容器

### 目标

用纯前端 Mock 地图替代真实百度地图，展示地图底图感觉和景点点位。

### 开发范围

- 创建地图容器组件。
- 使用 CSS 背景、网格、路线线条或简化标记模拟地图。
- 根据 Mock 景点数据展示点位。
- 点击点位更新当前选中景点。

### 验收标准

- 能看到地图区域和多个景点点位。
- 点击点位后有选中效果。
- 不加载百度地图 SDK。

### 教学重点

理解为什么地图先 Mock：避免第三方 API、Key、定位权限影响页面结构开发。

## 小模块 2.5：左侧景点推荐列表

### 目标

实现原型左侧“成都必玩推荐”列表。

### 开发范围

- 新建 `src/components/map-workbench/SpotRecommendList`。
- 展示景点名称、推荐理由、标签、评分、距离、建议游玩时间。
- 点击列表项更新 `selectedSpotId`，联动 Mock 地图点位。
- 先保留“推荐、距离、评分、最新”静态 Tab，只做选中态，不做真实排序。
- 先不使用真实图片，避免没有素材时出现坏图；封面图留到详情卡片或素材阶段统一处理。

### 当前实现文件

- `src/components/map-workbench/SpotRecommendList`
- `src/utils/map-workbench/spotDisplay.ts`
- `src/utils/map-workbench/spotFilters.ts`

### 验收标准

- 列表内容来自 Mock 数据。
- 当前选中项有明确视觉状态。
- 移动端列表区域不遮挡主要内容。
- `MapWorkbenchPage.tsx` 只传入数据和回调，不直接写列表 UI。

### 教学重点

理解列表组件拆分：列表只负责展示和选择，不负责决定数据从哪里来。

## 小模块 2.6：右侧景点详情卡片

### 目标

实现原型右侧景点详情区域。

### 开发范围

- 新建 `src/components/map-workbench/SpotDetailPanel`。
- 展示选中景点名称、评分、标签、简介、推荐理由。
- 展示开放时间、门票、建议游玩时间等信息。
- “加入行程”先接收回调，为 2.7 行程池做准备。
- “导航到这里”保留静态按钮，不接地图 API。
- 附近景点区域先展示同城其他 Mock 景点，点击可切换选中景点。

### 当前实现文件

- `src/components/map-workbench/SpotDetailPanel`
- `src/utils/map-workbench/spotDisplay.ts`

### 验收标准

- 点击不同景点后详情内容更新。
- 信息层级清晰，按钮状态明确。
- 小屏下详情区域能自然下移或折叠。
- 详情组件通过 Props 接收 `spot`，不直接导入 Mock 数据。

### 教学重点

理解“选中状态提升”：地图点位、左侧列表、右侧详情需要共享同一个 selectedSpotId。

## 小模块 2.7：底部行程池静态交互

### 目标

实现底部“我的行程”区域和加入/删除景点的基础交互。

### 开发范围

- 新建 `src/components/map-workbench/TripPlannerDock`。
- 在页面层维护 `tripSpotIds`，行程池根据 ids 显示已选择景点。
- 点击详情卡片的“加入行程”把当前景点加入池子。
- 支持删除单个景点、清空行程池。
- 起点、交通方式、规划模式先做受控表单，不调用路线规划。
- 暂不引入 Zustand，等跨页面或后端联动变复杂后再升级状态管理。

### 当前实现文件

- `src/components/map-workbench/TripPlannerDock`
- `src/pages/map-workbench/MapWorkbenchPage.tsx`

### 验收标准

- 同一个景点不能重复加入。
- 删除后页面状态正确。
- 底部区域在小屏下不遮挡所有内容。
- 不调用路线规划接口，不接百度地图。

### 教学重点

理解本地状态管理：这个阶段可以先用 React state，跨模块复杂后再引入 Zustand。

## 小模块 2.8：响应式与视觉整理

### 目标

统一页面间距、层级、滚动区域和移动端布局，为后续接接口和地图 API 打基础。

### 开发范围

- 优化桌面端布局比例。
- 优化移动端布局顺序。
- 检查文本溢出、按钮拥挤、内容遮挡。
- 统一颜色、阴影、圆角、字体大小。
- 将按钮、输入框、分段控件、标签、空状态、描述列表替换为 Ant Design 组件。
- 通过 `ConfigProvider` 统一 Ant Design 基础主题。

### 当前实现文件

- `src/App.tsx`
- `src/main.tsx`
- `src/components/map-workbench`
- `src/pages/map-workbench/MapWorkbenchPage.module.css`

### 验收标准

- 桌面端接近原型整体观感。
- 移动端可浏览、可点击、不横向溢出。
- `npm run lint` 和 `npm run build` 通过。

### 教学重点

理解响应式不是最后补丁，而是每个模块都要持续验证的约束。

## 阶段 2 完成标准

- 页面具备完整静态地图工作台结构。
- 主要交互包括：分类选中、景点选中、加入行程、删除行程。
- 所有业务数据来自 Mock。
- 不接后端接口，不接真实地图 API。
- 前端质量门禁全部通过。
