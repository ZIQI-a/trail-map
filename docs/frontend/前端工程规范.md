# 前端工程规范

## 技术栈

- React
- TypeScript
- Vite
- React Router
- Ant Design
- CSS Modules
- Zustand
- TanStack Query
- AntV L7
- 高德地图 JS API，第一阶段先 Mock

## 推荐目录结构

```plain
src
├── pages        路由级页面入口
├── assets       静态资源
├── components   页面组件和可复用组件
├── hooks        通用 Hooks
├── mocks        前端 Mock 数据
├── services     接口请求
├── stores       Zustand 状态
├── styles       全局样式
├── types        公共类型
└── utils        工具函数
```

## 目录职责

- `pages`：只放路由级页面，例如 `src/pages/map-workbench/MapWorkbenchPage.tsx`。页面负责组织布局、接收路由参数、维护少量跨组件共享状态。
- `components`：放页面组件和可复用组件。当前地图工作台组件统一放在 `src/components/map-workbench`。
- `mocks`：放前端 Mock 数据，例如 `src/mocks/map-workbench`。
- `utils`：放不依赖 UI 的工具函数，例如 Mock 地图坐标投影。
- `types`：放跨模块共享的数据结构，后续接口响应类型稳定后可继续拆细。

## 注释规范

- 组件函数需要添加简短中文注释，说明组件职责。
- 复杂交互、地图初始化、接口数据转换等重要步骤需要添加注释。
- 不写无意义注释，例如“给变量赋值”。

## 组件规范

- 页面组件只负责组织布局和调用业务组件。
- 不把顶部导航、地图、列表、详情、行程池等多个模块长期写在同一个页面文件中。
- 基础 UI 优先使用 Ant Design，例如 `Button`、`Input`、`Segmented`、`Tag`、`Empty`、`Descriptions`。
- CSS Modules 主要负责业务布局、地图浮层、间距和少量视觉覆盖，不重复造组件库已有控件。
- 通用组件避免绑定具体业务数据。
- 组件 Props 必须声明 TypeScript 类型。
- 状态优先放在最小需要范围内，跨组件共享时再使用 Zustand。

## 样式规范

- 业务组件优先使用 CSS Modules。
- 页面需要兼容 H5，小屏下不能出现横向溢出。
- 固定尺寸元素要设置合理的 `min-width`、`max-width` 或响应式规则。
- 交互元素要有清晰的 hover、focus、disabled 状态。

## 接口规范

- 所有接口请求统一放在 `src/services`。
- 后端返回结构变化时，同步更新 TypeScript 类型。
- 接口调用需要处理加载中、空数据和错误状态。
- 不在组件里拼接复杂接口参数，参数转换放在 service 或 utils 内部。

## 地图规范

- 第一阶段使用 Mock 地图容器，不接真实高德地图。
- 高德 Key 后续通过 `.env.local` 注入，禁止写死到源码。
- 地图相关组件集中放在 `src/components/map-workbench`，地图工具函数集中放在 `src/utils/map-workbench`，避免散落在页面各处。

## 教学式开发要求

- 每个模块开发完成后，说明文件位置、核心设计和运行验证方式。
- 对初学者不友好的抽象要谨慎使用，必要时在文档里解释原因。
- 如果发现需求与当前阶段目标冲突，应先提醒再继续开发。
- 默认不主动启动本地开发服务；页面预览由用户本地运行，AI 只做构建、lint 等验证。

## 质量门禁

- 每次功能改动后至少执行 `npm run lint` 和 `npm run build`。
- 如果改动涉及后端联调，需要同时确认后端接口或 Mock 数据契约。
- 如果质量门禁失败，不应视为改动完成。
- 除非用户明确要求，不执行 `npm run dev`。
