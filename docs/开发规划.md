# 全国旅游地图项目开发规划

## 阶段 1：项目初始化与规范文档

### 目标

从 0 建立前后端分离工程骨架，并把 AI 开发规范、阶段计划和地图 API 准备事项沉淀到文档中。

### 交付物

- `frontend/`：React + TypeScript + Vite 最小可运行项目。
- `backend/`：Spring Boot + Maven 最小可运行项目。
- `docs/`：文档索引、开发规划、地图 API 接入说明。
- `frontend/AGENT.MD`、`backend/AGENT.MD`：前后端 AI 开发规范。

### 验收标准

- 前端可以执行 `npm install` 和 `npm run build`。
- 后端可以执行 `mvn test`。
- 文档能说明后续每个阶段做什么、为什么这样做、如何验收。

### 教学说明

第一阶段先建立“能跑起来的空项目”。这样做的好处是后续每次只加一个小模块，出现问题时更容易定位。

前端用 Vite 是因为它启动快、配置少，适合 React 学习和中小型项目起步。后端用 Spring Boot 是因为它能快速搭建 REST API，后续和 MyBatis-Plus、MySQL、Knife4j 集成也比较自然。

## 阶段 2：前端静态地图工作台

### 目标

按照 `require/原型图.png` 搭建静态页面结构，先用 Mock 地图容器和模拟景点数据，不接真实后端。

### 交付物

- 顶部导航：品牌、搜索框、城市选择、操作按钮。
- 分类筛选：全部、自然风光、历史文化、城市地标等。
- 地图区域：先用静态容器模拟地图和点位。
- 左侧景点列表、右侧景点详情、底部行程池。

### 验收标准

- 页面在桌面端接近原型结构。
- 小屏宽度下内容不溢出，为 H5 兼容打基础。
- 所有数据来自前端 Mock 文件，便于后续替换为接口数据。

### 教学说明

先做静态页面可以避免“样式、状态、接口、地图 API”同时混在一起。等页面结构稳定后，再逐步接入后端接口和地图能力。

### 细分计划

阶段 2 不一次性完成，按 `docs/frontend/阶段2-前端静态地图工作台拆分计划.md` 拆成多个小模块逐步开发。每个小模块完成后先审查，再进入下一个小模块。

## 阶段 3：后端城市与景点基础接口

### 目标

建立城市、景点、标签相关的基础数据模型和查询接口，为前端展示景点列表与详情提供数据。

### 交付物

- 数据表优先级第一批：`city`、`spot`、`spot_tag`、`spot_tag_relation`。
- 基础接口：城市列表、城市详情、景点列表、景点详情、标签筛选。
  列表接口支持可选分页参数，未传时默认返回全部数据。
- Flyway 数据库迁移脚本与初始化数据。
- Knife4j / Swagger 接口文档。

### 验收标准

- 前端可以通过接口获取城市和景点数据。
- 后端接口返回结构统一，错误信息清晰。
- 数据库可通过 Flyway 自动完成初始化和后续迁移。
- 数据库字段与 `require/数据库设计.md` 保持一致，必要调整需同步文档。

### 教学说明

这一阶段重点学习 Spring Boot 分层结构：Controller 接收请求，Service 处理业务逻辑，Mapper 访问数据库，Entity 对应数据表。
同时引入 Flyway 管理表结构和初始化数据，避免后续部署、测试和上线环境各自维护一套 SQL。

## 阶段 4：前后端联调

### 目标

把第二阶段的 Mock 数据替换为后端真实接口，完成城市切换、景点筛选和景点详情展示。

### 交付物

- TanStack Query 请求封装。
- 前端接口类型定义。
- 城市、标签、景点列表和景点详情接口联调。
- 页头城市切换与联动刷新。
- 基础加载态、错误态和空数据态。
- Vite 本地代理配置，避免开发阶段把后端地址写死到页面代码。

### 验收标准

- 前端不再依赖静态 Mock 作为主数据源。
- 后端服务停止时，前端能显示友好的错误状态。
- 切换城市后，景点列表、标签和详情能同步刷新。
- 筛选条件变化后能正确刷新景点列表。

### 教学说明

这一阶段重点学习前后端分离：前端只关心接口契约，后端只保证数据和业务规则正确。
联调阶段建议把请求层、查询层和页面展示层拆开，后续接入分页、地图 API 或城市切换时改动会更可控。

## 阶段 5：地图 API 接入

### 目标

接入百度地图 JavaScript API，替换 Mock 地图容器，展示真实地图和景点点位。

### 交付物

- 百度地图 AK 配置说明。
- 地图加载、城市定位、缩放、点位 Marker。
- 景点点击联动右侧详情。
- 区域型景点支持轮廓高亮展示。
- 后端提供 POI 校准候选接口，辅助修正精选景点主点位。
- 精选景点坐标校准结果通过独立 Flyway 版本持续沉淀。
- 批量校准/批量导入工具先记入后续事项，当前阶段仍以少量精选景点校准为主。
- 地图 Key 不提交到代码仓库，使用环境变量管理。

### 验收标准

- 地图能正常加载到所选城市。
- 景点点位位置正确。
- 点击地图 Marker 后，右侧详情与左侧列表选中状态能同步联动。
- 城市默认展示全景，选中景点后地图能中等缩放查看具体位置。
- 区域型景点在选中后可展示轮廓范围，没有轮廓时回退到主点位。
- 没有把真实 Key 写死到源码中。

### 教学说明

地图 API 属于第三方能力，接入前需要先申请 AK，并配置安全域名。Key 管理要从一开始规范，否则后期容易泄露。

## 阶段 6：路线规划 MVP

### 目标

实现“景点池 + 自由路线模式”的第一版路线规划。完整行程模式暂不作为默认能力。

### 交付物

- 景点加入行程池、删除、清空。
- 出发地、交通方式、规划模式选择。
- 行程规划请求/响应模型，统一表达景点顺序、路线分段、停留时间和总行程时间。
- 后端路线规划接口先以自由路线模式落地，景点顺序暂按景点池顺序执行。
- 当前交通方式先收敛为公共交通、驾车、步行、骑行；打车单独能力后续再评估。
- 起点输入支持地点联想候选和当前位置快捷填充；若未选中候选项，则点击规划时再尝试地点检索解析坐标，失败时回退到城市中心点。
- 前端行程池提交规划参数并展示路线摘要、分段时间和总行程时间。
- 后端路线规划记录和路线分段保存。
- 调用百度地图路线规划能力生成分段路线。

### 验收标准

- 用户选择多个景点后可以生成路线顺序。
- 每段路线展示交通方式、耗时、距离和说明。
- 返回结果能区分交通时间、景点停留时间和总行程时间。
- 后端能保存 `route_record` 和 `route_segment`。

### 教学说明

路线规划不要一开始就做完整日程。先把“自由路线模式”的数据结构和时间口径做稳：景点顺序、每段路线、建议停留时长、总交通时间、总游玩时间。等这层稳定后，再扩展多天拆分和时间表。

## 阶段 6.5：完整行程模式

### 目标

在自由路线模式稳定后，补“完整行程模式”的第一版时间编排能力，输出 `Day 1 / Day 2` 这种可直接阅读的多天时间表。当前阶段先做“用户半指定、系统半编排”，不直接做酒店推荐、餐厅推荐和保姆级自动攻略。

### 分层范围

#### 第一层：完整行程模式 V1（当前优先）

- 用户自己选择景点、起点或酒店地点。
- 用户选择游玩天数、每天开始时间、每天结束时间、行程强度。
- 系统基于现有路线顺序、交通耗时和景点建议停留时间，自动拆分成多天。
- 系统可插入固定的午餐时间块或休息时间块，但不先做餐厅推荐。
- 前端按 `Day 1 / Day 2` 展示每日时间轴。

#### 第二层：酒店接入（V1 后）

- 支持用户手动输入或联想选择酒店/住宿地点。
- 支持“每天从酒店出发，晚上回酒店”的时间编排。
- 不在这一层自动推荐酒店，只先接入用户自定义地点。

#### 第三层：用餐与攻略增强（后置）

- 根据时间块插入午餐、晚餐、休息窗口。
- 后续再评估餐厅推荐、酒店推荐、夜游建议、雨天替代方案等增强能力。
- 这部分属于产品增强，不作为完整行程模式的第一版阻塞项。

### 当前不做

- 自动推荐酒店。
- 自动推荐餐厅。
- AI 文案式“保姆级攻略”。
- 多套备选方案自动生成。
- 完整的运营后台配置能力。

### 交付物

- 完整行程模式的输入模型补充：
  - `tripDays`
  - `dailyStartTime`
  - `dailyEndTime`
  - `intensity`
  - `includeLunchBreak`
  - `includeNightTour`
  - 可选 `hotel/startPoint`
- 后端时间编排器：
  - 按天拆分景点
  - 叠加交通耗时和景点停留时长
  - 生成到达/离开时间
  - 插入固定午餐时间块
- 响应模型中的 `itineraryDays` 真正落地。
- 前端路线抽屉升级为按天展示。

### 当前实现状态

- 后端已能在 `schedule` 模式下生成 `itineraryDays`。
- 当前先采用默认时间参数兜底：默认 2 天、09:00-18:00、标准强度。
- 前端已补“完整行程配置弹窗”，用户在点击开始规划后可先填写天数、作息、强度和午餐开关。
- 生成完整行程后，页面改为“地图上方 Day 标签 + 右侧当天详细行程 + 悬浮设置按钮”的骨架，右侧详细行程继续复用路线抽屉思路。
- 酒店、午餐、休息节点已开始接入完整行程编排：
  - 用户可选择“系统推荐 / 手动输入 / 不安排”
  - 手动输入已补为联想候选，优先展示更靠近当前景点池的 POI
  - 如果用户只输入文本、没有选中候选，规划提交前仍会兜底做 POI 检索解析坐标
  - 系统推荐优先按当前景点附近实时检索商场、餐饮、酒店等 POI
  - 未配置服务端 AK 或未命中候选时，会回退为普通时间块，不阻塞完整行程主流程

### 验收标准

- 用户选择 2 天或 3 天后，可以得到按天拆分的行程。
- 每天都能展示出发时间、景点到达时间、停留时间、离开时间。
- 返回结果能够区分每日交通时间、每日游玩时间和每日总时长。
- 午餐、休息、酒店节点可以按规则插入；有推荐或手动地点时可展示真实地点，否则回退为普通时间块。

### 教学说明

完整行程模式不要一开始就做“全自动旅行管家”。正确顺序是：

- 先做多天时间拆分和时间表。
- 再做酒店地点接入。
- 最后再考虑餐饮推荐、攻略增强和更复杂的偏好系统。

这样可以保证路线规划核心逻辑、时间编排逻辑、推荐逻辑彼此解耦，便于测试和后续扩展。

## 阶段 7：用户系统与旅行足迹

### 目标

实现登录、收藏、打卡、隐藏已去景点等二期能力。

### 拆分计划

阶段 7 按 `docs/backend/阶段7-用户管理拆分计划.md` 逐步推进。第一步只建立用户域基础表和后端实体，不直接开发完整登录注册页面。

### 当前实现状态

- 已确认路线规划和完整行程主流程可以支撑进入用户系统阶段。
- 用户表实际命名为 `app_user`，避免使用 `user` 带来的数据库保留语义和运维歧义。
- 用户域基础表通过 Flyway `V8__create_user_domain_tables.sql` 管理。
- 已实现用户基础 CRUD、注册、登录和当前用户查询。
- 当前登录令牌仍使用轻量 HMAC Bearer Token，但鉴权层已接入 Spring Security。
- `/api/users/**` 管理接口当前仅允许 `admin` 角色访问。
- 前端已接入顶部登录入口：未登录展示登录按钮，登录后展示用户昵称，退出时清理本地 token。
- 前端请求层会自动携带 `Authorization: Bearer token`，后续收藏、打卡、保存行程接口可直接复用。
- 已实现收藏接口：`GET /api/favorite-spots/{spotId}/status`、`POST /api/favorite-spots/{spotId}`、`DELETE /api/favorite-spots/{spotId}`。
- 景点详情面板已接入收藏按钮，未登录点击会先拉起登录弹窗。
- 已新增独立“我的收藏”页面 `/favorites`，当前先展示景点收藏，并从首页顶部收藏夹和个人头像二级菜单进入。
- 个人打卡模块已进入阶段 7.6，第一版不做定位强校验，先实现“用户主动打卡/取消打卡/我的足迹列表”的后端闭环。
- “我的行程”列表的城市筛选、行程类型筛选、排序和分页已下沉到后端，前端不再拉取大批量数据后本地二次过滤。

### 交付物

- 用户注册登录。
- 收藏景点、打卡景点。
- 去过的景点变灰或隐藏。
- 城市完成度和旅行足迹基础展示。

### 验收标准

- 用户数据与匿名路线数据分离。
- 登录前后的页面状态清晰。
- 打卡和收藏不影响公共景点数据。

### 教学说明

用户系统会引入认证、权限和数据归属问题，因此放到地图和景点主流程稳定之后再做。
