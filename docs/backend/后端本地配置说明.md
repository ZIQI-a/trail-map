# 后端本地配置说明

本文档只回答一个问题：为了让 `backend/` 能在你本机正常启动，你需要手动做哪些事。

## 1. 先理解几个名词

- `MySQL`：后端实际连接的数据库服务。
- `trailmap`：你已经创建好的数据库名。
- `环境变量`：启动程序前提供给 Spring Boot 的外部配置，避免把账号密码写死到代码里。
- `Flyway`：数据库迁移工具。应用启动时会自动检查并执行 `backend/src/main/resources/db/migration/` 下的 SQL 文件。
- `flyway_schema_history`：Flyway 自动创建的版本记录表，用来记住哪些 SQL 已经执行过，不需要手动创建。

## 2. 你现在已经完成了什么

- 你已经创建好数据库 `trailmap`。

这意味着你不需要再手动执行建表 SQL 了。后端第一次启动时，Flyway 会自动建表并插入初始化数据。

## 3. 你还需要手动做什么

### 3.1 确认 MySQL 服务正在运行

你需要保证本机 MySQL 已启动，并且你知道这几个信息：

- 数据库地址：通常是 `localhost`
- 端口：通常是 `3306`
- 用户名：例如 `root`
- 密码：你自己设置的密码

### 3.2 配置环境变量或本地 `.env.local`

后端现在支持像前端一样，在 `backend/.env.local` 中放本地专用配置。  
对应加载配置见 [application.yml](/Users/Code/SelfCode/TrailMap/backend/src/main/resources/application.yml:1)，启动时会自动导入这个文件。

推荐优先方式：

1. 把只属于你本机的配置写进 `backend/.env.local`
2. 不要提交这个文件到仓库

例如：

```plain
BAIDU_MAP_SERVER_AK=你的百度地图服务端AK
TRAILMAP_DB_HOST=localhost
TRAILMAP_DB_PORT=3306
TRAILMAP_DB_NAME=trailmap
TRAILMAP_DB_USERNAME=root
TRAILMAP_DB_PASSWORD=你的数据库密码
TRAILMAP_AUTH_TOKEN_SECRET=本地登录令牌密钥
```

如果你不想用 `.env.local`，仍然可以继续用环境变量，因为数据库账号密码不适合长期写进仓库配置文件。

后端现在从环境变量读取数据库连接信息，对应代码见 [application.yml](/Users/Code/SelfCode/TrailMap/backend/src/main/resources/application.yml:1)。

在终端里执行：

```bash
export TRAILMAP_DB_HOST=localhost
export TRAILMAP_DB_PORT=3306
export TRAILMAP_DB_NAME=trailmap
export TRAILMAP_DB_USERNAME=root
export TRAILMAP_DB_PASSWORD=你的数据库密码
export TRAILMAP_AUTH_TOKEN_SECRET=本地登录令牌密钥
```

说明：

- `TRAILMAP_DB_HOST`：MySQL 主机地址，本机一般填 `localhost`
- `TRAILMAP_DB_PORT`：MySQL 端口，默认 `3306`
- `TRAILMAP_DB_NAME`：数据库名，你这里就是 `trailmap`
- `TRAILMAP_DB_USERNAME`：数据库用户名
- `TRAILMAP_DB_PASSWORD`：数据库密码
- `TRAILMAP_AUTH_TOKEN_SECRET`：登录令牌签名密钥，本地开发可随意设置，上线必须换成高强度随机值

如果你关闭当前终端窗口，这组变量会失效。下次重新打开终端后，需要重新执行一次，或者把它们写进你的 shell 配置文件。

如果你已经把账号密码直接写进本地 `application.yml`，项目也可以启动，但要注意：

- 这种方式只适合你自己的本地环境。
- 如果该文件会提交到仓库，就有泄露风险。
- 更稳妥的做法是把默认值留空或占位，本地通过环境变量覆盖。

如果你要使用“POI 校准候选接口”或“路线规划接口”调用百度服务端能力，还需要额外配置：

```bash
export BAIDU_MAP_SERVER_AK=你的百度地图服务端AK
```

说明：

- 这个 AK 用于后端调用百度地点检索，不是前端浏览器端 AK。
- 未配置 `BAIDU_MAP_SERVER_AK` 时，POI 校准接口不可用，路线规划接口会退回本地估算路线。

### 3.3 启动后端

```bash
cd /Users/Code/SelfCode/TrailMap/backend
mvn spring-boot:run
```

项目已经在 Maven 配置里补了 `--enable-native-access=ALL-UNNAMED`，用于兼容较新的 JDK 在启动嵌入式 Tomcat 时对原生访问的限制提示。

因此通过下面这些方式启动时，不需要你再额外手动加这个参数：

- `mvn spring-boot:run`
- `mvn test`
- 其他在 `backend/` 目录下执行的 Maven 命令

如果你不是通过 Maven，而是直接在 IntelliJ IDEA 里运行 `TrailMapApplication` 主类，则需要在 Run Configuration 里手动补这个 VM 选项：

```plain
--enable-native-access=ALL-UNNAMED
```

第一次启动时，Flyway 会自动执行迁移脚本：

- [V1__init_core_tables.sql](/Users/Code/SelfCode/TrailMap/backend/src/main/resources/db/migration/V1__init_core_tables.sql:1)

执行成功后，`trailmap` 数据库里会出现：

- `city`
- `spot`
- `spot_tag`
- `spot_tag_relation`
- `flyway_schema_history`

## 4. 如何判断配置是否成功

启动成功后，访问：

- 健康检查：[http://localhost:8080/api/health](http://localhost:8080/api/health)
- 接口文档：[http://localhost:8080/swagger-ui.html](http://localhost:8080/swagger-ui.html)
- POI 校准候选接口示例：
  [http://localhost:8080/api/poi-calibration/candidates?cityName=西安市&keyword=大唐不夜城&addressKeyword=慈恩路](http://localhost:8080/api/poi-calibration/candidates?cityName=西安市&keyword=大唐不夜城&addressKeyword=慈恩路)

如果能正常打开，说明数据库连接和 Flyway 迁移都已经成功。

## 5. 如果启动失败，优先检查什么

### 5.1 用户名或密码错误

常见现象：

- 启动日志里出现数据库认证失败

处理方式：

- 检查 `TRAILMAP_DB_USERNAME`
- 检查 `TRAILMAP_DB_PASSWORD`

### 5.2 数据库服务没启动

常见现象：

- 连接超时
- 拒绝连接

处理方式：

- 确认 MySQL 已启动
- 确认端口确实是 `3306`

### 5.3 数据库名不对

常见现象：

- 提示数据库不存在

处理方式：

- 确认 `TRAILMAP_DB_NAME=trailmap`

### 5.4 Flyway 执行报错

常见现象：

- 启动时提示某条迁移 SQL 执行失败

处理方式：

- 先不要手动删表
- 先看报错对应的是哪条 SQL
- 如果是你本地库里已有半成品旧表，需要先确认这些旧表是否要保留

如果数据库里已经有人为创建过同名表，但结构和当前迁移脚本不一致，Flyway 会失败。这种情况下不要盲目重试，先处理旧表状态。

### 5.5 JDK 原生访问警告

常见现象：

```plain
WARNING: A restricted method in java.lang.System has been called
WARNING: java.lang.System::load has been called by org.apache.tomcat.jni.Library
```

说明：

- 这是较新版本 JDK 对嵌入式 Tomcat 原生加载行为的提示，不是你的业务代码报错。
- 项目已经在 Maven 启动链路里处理了这个问题。

处理方式：

- 如果你用 `mvn spring-boot:run` 启动，一般不需要再处理。
- 如果你用 IDEA 直接运行主类，请在 VM options 中加上：

```plain
--enable-native-access=ALL-UNNAMED
```

## 6. 后续新增表或字段时怎么做

不要直接改线上库，也不要手动在本地库里加字段后不留记录。

正确做法是：

1. 在 `backend/src/main/resources/db/migration/` 下新增一个迁移文件
2. 文件名按 Flyway 规范递增，例如：
   - `V2__add_travel_area_table.sql`
   - `V3__add_spot_image_table.sql`
3. 启动应用，让 Flyway 自动执行

这样本地、测试、上线环境都能按同一套迁移历史推进。

下一步你只需要：

1. 配置环境变量
2. 启动后端
3. 检查 Flyway 是否自动建表成功
