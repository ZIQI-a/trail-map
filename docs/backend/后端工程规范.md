# 后端工程规范

## 技术栈

- Java
- Spring Boot
- Maven
- MyBatis-Plus
- MySQL
- Flyway
- Knife4j / Swagger
- Redis，后期按需加入

## 推荐包结构

```plain
src/main/java/com/trailmap
├── common       通用返回、异常、常量
├── config       Spring 配置
├── controller   REST 接口
├── entity       数据库实体
├── mapper       MyBatis-Plus Mapper
├── service      业务接口与实现
└── TrailMapApplication.java
```

## 注释规范

- Controller、Service、重要方法需要添加中文注释，说明职责。
- 复杂业务规则、第三方地图 API 参数转换、路线规划逻辑必须写注释。
- 注释要解释“为什么这样做”，避免重复代码本身。

## 分层规范

- Controller 只负责参数接收、基础校验和返回结果。
- Service 负责业务逻辑，不直接暴露数据库细节给 Controller。
- Mapper 只负责数据库访问，不写业务判断。
- Entity 字段应和数据库设计保持一致，字段调整必须同步文档。
- Entity 统一使用 Lombok `@Getter` 和 `@Setter` 生成存取方法；不默认使用 `@Data`，避免实体的 `equals`、`hashCode` 和 `toString` 语义被隐式改变。
- 地图相关实体如果增加 `boundary`、坐标系、POI 来源等字段，要明确“主点位”和“轮廓范围”的业务含义。

## 接口规范

- URL 使用 `/api` 前缀。
- 返回结构后续统一封装，避免每个接口格式不一致。
- 异常信息要清晰，不能把底层堆栈直接返回给前端。
- 接口入参需要校验，尤其是分页、城市 ID、景点 ID、经纬度等字段。
- 列表接口如果支持分页，应统一使用 `pageNum`、`pageSize` 参数和统一分页响应结构。
- 当分页参数都不传时，列表接口默认返回全部数据；只传一个参数时，另一个参数走后端默认值。
- 管理端首页这类跨领域统计应优先提供聚合接口，例如 `/api/admin/overview`，不要让前端为了概览页拉取用户、城市、景点全量列表后自行统计。

## 数据库规范

- MVP 第一批表：`city`、`spot`、`spot_tag`、`spot_tag_relation`。
- 第二批表：`travel_area`、`travel_area_spot`、`route_record`、`route_segment`。
- 用户系统相关表放到二期，用户主表实际使用 `app_user`，避免直接使用 `user` 表名。
- 枚举类字段可以先用字符串落库，后端用枚举集中管理。
- 表结构和初始化数据优先通过 Flyway 迁移脚本维护，不再依赖手工执行零散 SQL。
- 新增表、字段、索引或初始化数据时，应新增 Flyway 版本脚本，不要直接改线上或本地数据库后不留记录。
- 已提交或已执行的 Flyway `V*` 迁移脚本视为不可变历史，禁止直接修改内容；发现问题必须新增后续版本修正。
- 推荐把“建表”和“初始化基础数据”拆成独立迁移版本，例如 `V1` 建表、`V2` 种子数据，降低回滚和排查成本。
- 需要整理历史迁移时，只能补充迁移说明文档或在明确允许重建全部环境时做基线压缩，不能在普通开发中直接重写旧版本。
- 使用 MySQL 时，Flyway 依赖应同时包含 `flyway-core` 和 `flyway-mysql`，避免启动时出现数据库版本不支持。
- 区域型景点如果需要高亮轮廓，可在景点表存可选边界数据；轮廓只是展示增强，不替代景点主点位。

## 文档同步规范

- 更新业务代码时，必须同步检查相关说明文档是否需要更新。
- 涉及接口变更时，同步更新接口说明或阶段规划中的相关描述。
- 涉及数据库结构、迁移方式、初始化方式变更时，同步更新数据库设计文档、后端本地配置说明和相关索引。
- 如果当前实现已切换为 Flyway、MyBatis-Plus、分页等机制，相关文档必须明确写出，不能继续保留过期说明。

## 地图 API 规范

- 服务端 Key 后续通过环境变量或安全配置注入，禁止写死在代码中。
- 调用高德路线规划接口时，要保留原始请求和响应，便于调试和缓存。
- 第三方接口失败时，返回可理解的业务错误，不让前端直接面对原始异常。

## 教学式开发要求

- 每次新增模块时，说明它在 Spring Boot 分层结构中的位置。
- 对 MyBatis-Plus、注解、配置项等内容，提供简短解释。
- 如果发现数据库设计不适合当前需求，应先说明原因，再做最小调整。

## 质量门禁

- 后端功能改动后至少执行 `mvn test`。
- 如果改动影响前端联调，需要补充执行前端 `npm run build`。
- 如果质量门禁失败，不应视为改动完成。
