# 阶段 7：用户管理与旅行足迹拆分计划

## 当前结论

路线规划和完整行程主流程已经具备继续扩展用户能力的基础。用户管理会引入账号归属、登录态、收藏、打卡和行程保存，因此不一次性完成，按小模块推进。

## 表名调整说明

数据库草案中原计划使用 `user` 表。实际开发中改为 `app_user`，避免和数据库系统用户、保留语义或运维工具产生歧义。后续代码实体命名为 `AppUser`。

## 7.1 用户域基础

### 目标

先建立用户系统的数据库和后端基础结构，不开放登录注册接口。

### 交付物

- Flyway 迁移脚本创建：
  - `app_user`
  - `user_favorite_spot`
  - `user_checkin_spot`
  - `user_trip`
  - `user_trip_spot`
- 后端新增 `AppUser` 实体和 `AppUserMapper`。
- 同步数据库设计文档。

### 验收标准

- `mvn test` 可以通过。
- 用户表字段支持后续注册、登录、头像、手机号、邮箱、账号状态和用户类型。
- 收藏、打卡、保存行程表已经有唯一约束，避免重复数据。

### 教学说明

先建表和实体，是为了让后续注册登录接口有稳定的数据模型。这个阶段不引入认证框架，避免“表结构、密码加密、登录态、前端页面”同时开发导致问题难定位。

## 7.2 注册与登录接口

### 目标

实现最小账号能力：用户名注册、密码登录、当前用户信息查询。

### 交付物

- 注册接口：`POST /api/auth/register`
- 登录接口：`POST /api/auth/login`
- 当前用户接口：`GET /api/auth/me`
- 密码使用哈希存储，不保存明文。
- 返回前端可保存的登录令牌。
- 用户基础管理接口：
  - `GET /api/users`
  - `GET /api/users/{userId}`
  - `POST /api/users`
  - `PUT /api/users/{userId}`
  - `DELETE /api/users/{userId}`

### 当前实现状态

- 已实现用户基础 CRUD。
- 已实现注册、登录和当前用户查询。
- 密码使用 PBKDF2 哈希存储。
- 登录令牌采用轻量 HMAC Bearer Token，签名密钥通过 `TRAILMAP_AUTH_TOKEN_SECRET` 配置。
- 当前已接入 Spring Security 无状态鉴权链，继续复用现有 HMAC Bearer Token。
- `/api/users/**` 当前仅允许 `admin` 角色访问，普通用户访问返回 403。
- 由于注册入口默认只创建 `normal` 用户，首个管理员账号需要通过数据库初始化或运维脚本提升角色。
- 已实现收藏接口：
  - `GET /api/favorite-spots/{spotId}/status`
  - `POST /api/favorite-spots/{spotId}`
  - `DELETE /api/favorite-spots/{spotId}`
- 收藏接口要求登录访问，仅操作当前用户自己的收藏关系。
- 已开始阶段 7.6 个人打卡模块，详细计划见 `docs/backend/阶段7.6-个人打卡模块拆分计划.md`。

### 验收标准

- 用户名不可重复。
- 密码不以明文落库。
- 新注册用户默认类型为 `normal`，后续再扩展 `member`、`admin` 权限规则。
- 未登录访问需要登录的接口时返回明确错误。
- `mvn test` 覆盖注册、登录、当前用户校验和用户 CRUD。

## 7.3 前端登录入口

### 目标

把顶部头像从静态展示升级为登录入口。

### 交付物

- 登录/注册弹窗。
- 登录成功后展示用户昵称或头像。
- 前端保存登录态，并在请求中携带令牌。

### 验收标准

- 未登录时仍可浏览地图和规划路线。
- 登录后能访问用户相关能力。
- 退出登录后清空本地登录态。

## 7.4 收藏与打卡

### 目标

实现景点收藏、打卡去过、隐藏已去景点。

### 交付物

- 收藏/取消收藏接口。
- 打卡/取消打卡接口。
- 景点列表叠加当前用户的收藏和打卡状态。
- 前端景点卡片、详情面板展示收藏与打卡按钮。

### 验收标准

- 收藏和打卡只影响当前用户，不改变公共景点数据。
- 已打卡景点可以变灰或隐藏。
- 重复收藏、重复打卡不会产生重复数据。

### 当前实现状态

- 收藏接口和前端收藏入口已完成。
- 打卡后端第一版按阶段 7.6 推进，不做定位距离校验，只记录用户主动打卡关系、可选坐标和备注。

## 7.5 保存行程

### 目标

把自由路线和完整行程结果保存到用户账号下。

### 交付物

- 保存行程接口。
- 我的行程列表。
- 行程详情回放。

### 验收标准

- 保存行程必须登录。
- 能区分自由路线和完整行程。
- 后续可关联路线记录和每天的景点顺序。
