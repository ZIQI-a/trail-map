# 数据库迁移版本说明

## 核心原则

Flyway 的 `V*` 版本脚本一旦进入共享分支、被本地或线上数据库执行，就视为不可变历史。后续发现问题时，不允许直接修改已存在的 `V1`、`V2`、`V9` 等脚本内容，必须新增下一个版本脚本修正。

原因是 Flyway 会记录每个迁移文件的 checksum。修改已执行脚本会导致数据库启动时报 checksum mismatch，严重时会让不同开发环境、测试环境和生产环境的结构不一致。

## 正确处理方式

- 新增表、字段、索引、约束：新增 `V{下一个版本}__xxx.sql`。
- 修复历史字段设计：新增修复迁移，例如 `V10__fix_user_trip_item_schema.sql`。
- 修正初始化数据：新增数据修复迁移，例如 `V11__patch_spot_seed_data.sql`。
- 需要重建干净结构：只能在项目尚未正式部署前，经团队确认后做一次“基线重建”；已部署后不能覆盖历史版本。
- 测试库兼容问题：不要修改正式迁移文件，可使用测试专用配置或新增兼容迁移策略单独处理。

## 当前版本分类

### 核心基础结构

- `V1__init_core_tables.sql`
  - 创建城市、景点、景点标签、景点标签关系四张 MVP 核心表。

### 初始化数据

- `V2__seed_base_data.sql`
  - 初始化成都、西安、基础标签和首批精选景点数据。

### 景点展示增强

- `V3__add_spot_boundary.sql`
  - 给 `spot` 增加 `boundary_geojson`，用于区域型景点轮廓展示。

### 景点坐标校准

- `V4__calibrate_xian_spots.sql`
  - 初步校准西安景点主点位和示例边界。
- `V5__refine_xian_spots_by_baidu_place_api.sql`
  - 基于百度地点检索进一步校准西安景点，并清理不准确边界。
- `V6__calibrate_all_spots_by_baidu_place_api.sql`
  - 统一校准成都和西安景点主点位，当前阶段清空边界数据。

### 景点媒体数据

- `V7__replace_spot_cover_urls_with_real_images.sql`
  - 将首批景点封面图替换为可访问的外部图片链接。

### 用户域

- `V8__create_user_domain_tables.sql`
  - 创建 `app_user`、`user_favorite_spot`、`user_checkin_spot`、`user_trip`、`user_trip_spot`。
  - 当前个人打卡模块复用这里已有的 `user_checkin_spot` 表。

### 路线与行程持久化

- `V9__add_route_persistence_and_refine_trip.sql`
  - 创建 `route_record`、`route_segment`。
  - 将 `user_trip_spot` 升级为更通用的 `user_trip_item`。
  - 给 `user_trip` 补充起终点和汇总耗时字段。

## 后续建议

### 短期

- 不再修改 `V1` 到 `V9`。
- 如果发现 V9 对某些测试数据库语法不兼容，新增测试兼容策略，不能改 V9。
- 个人打卡当前不需要新迁移，后续如果要增加打卡图片、心情、同行人等字段，再新增版本。

### 中期

- 可以新增管理端数据维护迁移，例如城市扩充、景点扩充、标签扩充。
- 可以新增统计字段或索引迁移，例如给 `user_checkin_spot(user_id, checkin_time)` 增加索引。

### 项目正式发布前可选整理

如果项目尚未发布，且所有开发数据库都可以重建，可以讨论做一次“基线压缩”：

1. 备份现有 `db/migration`。
2. 生成新的干净基线脚本，例如 `V1__baseline_schema.sql`。
3. 清空开发库，重新执行 Flyway。
4. 所有人同步新基线。

这个动作会重写历史，不适合已部署项目；没有明确确认前不要执行。
