# 行迹旅图 - 全栈偏前端项目简历写法

## 项目定位

这份写法适合投递“全栈开发（偏前端）”“前端开发但要求懂后端”“Web 开发工程师”等岗位。表达重点是：前端能力是主线，后端能力能支撑完整业务闭环。

不要把自己包装成纯后端资深工程师。更合适的定位是：

```text
以前端复杂交互和地图可视化为主，同时能够独立完成 Spring Boot 接口、数据库建模、认证授权和前后端联调。
```

## 技术栈写法

```text
前端：React 19、TypeScript、Vite、React Router、Ant Design、TanStack Query、CSS Modules、百度地图 JavaScript API、ECharts
后端：Java 21、Spring Boot 3、MyBatis-Plus、MySQL、Flyway、Spring Security、SpringDoc OpenAPI
```

如果简历空间有限：

```text
React + TypeScript + 百度地图 API + Spring Boot + MyBatis-Plus + MySQL + Flyway
```

## 精简版

```text
行迹旅图 - 全国旅游地图规划平台
技术栈：React、TypeScript、TanStack Query、百度地图 JS API、Spring Boot、MyBatis-Plus、MySQL、Flyway

- 完成城市景点浏览、路线规划、完整行程、收藏打卡、我的行程和公开分享等核心链路，形成从地图浏览到数据沉淀的业务闭环。
- 前端处理地图工作台多区域联动与路线可视化；后端提供城市、景点、路线规划、用户行程等 RESTful API 支撑页面交互。
- 设计并维护 MySQL 核心表结构，使用 Flyway 管理数据库迁移，覆盖 city、spot、route_record、route_segment、app_user、user_trip 等业务表。
- 接入 Spring Security 和 HMAC Bearer Token，实现普通登录态接口、公开分享访问和管理员接口权限控制。
```

## 优化版

```text
行迹旅图 - 全国旅游地图规划平台
项目描述：前后端分离的旅游地图规划系统，用户可以在地图中浏览城市景点，生成自由路线或多天完整行程，并管理个人收藏、打卡、保存行程和公开分享。
技术栈：React 19、TypeScript、Vite、TanStack Query、Ant Design、百度地图 JS API、Java 21、Spring Boot 3、MyBatis-Plus、MySQL、Flyway、Spring Security

- 完成“景点发现-行程规划-路线可视化-行程保存/分享”的功能闭环，前端重点处理地图、列表、详情、行程池和路线结果之间的复杂联动。
- 前端基于 TanStack Query 管理城市、景点、标签、POI 候选、路线规划、收藏/打卡状态和行程详情等服务端状态，减少组件内异步逻辑堆积。
- 封装百度地图 JS API 接入，包括脚本单例加载、坐标转换、Marker / Polyline 渲染、路线方向标识、地图选点和当前位置展示。
- 后端基于 Spring Boot + MyBatis-Plus 提供 RESTful API，包含城市景点、路线规划、用户认证、收藏打卡、我的行程、公开分享和后台管理模块。
- 使用 Flyway 管理数据库版本，围绕城市、景点、标签、路线记录、路线分段、用户、收藏、打卡、行程节点等表设计数据模型和约束。
- 在认证链路上区分普通登录态、公开分享态和管理员权限，后端通过过滤器与角色校验保护接口，前端请求层统一注入 Authorization 头。
```

## 面试官友好版

```text
行迹旅图 - 全国旅游地图规划平台
项目描述：一个偏实际业务场景的全栈项目。前端负责地图工作台、个人中心和后台管理页面；后端负责 REST API、路线规划服务、用户体系和数据持久化。项目重点解决旅游景点地图浏览、真实路线规划和用户行程沉淀问题。
技术栈：React、TypeScript、TanStack Query、Ant Design、百度地图 API、Spring Boot、MyBatis-Plus、MySQL、Flyway、Spring Security

- 前端侧解决地图工作台的复杂联动问题，完成城市切换、景点筛选、地图点位、详情联动、行程池、路线结果抽屉、完整行程 Day 时间轴和公开分享回放。
- 后端侧实现城市/景点/标签查询、路线规划、POI 候选、用户认证、收藏打卡、用户行程和管理后台接口，统一使用 ApiResponse 返回结构。
- 路线规划服务接入百度地图服务端 API，按起点和景点顺序生成路线分段；当未配置 AK 或第三方调用失败时返回本地估算结果，保证主流程不中断。
- 数据层使用 MyBatis-Plus + MySQL，Flyway 管理迁移脚本，核心表覆盖景点数据、路线记录、用户域和行程节点，支持后续功能演进。
- 认证授权采用 HMAC Bearer Token + Spring Security，区分普通登录态、公开分享访问和管理员接口保护。
```

## 全栈偏前端的讲法

面试时推荐这样讲：

```text
我这个项目主要以前端为主，最复杂的是地图工作台和路线展示。但为了让项目不是纯静态页面，我也做了后端接口和数据库。后端主要用 Spring Boot 提供 REST API，用 MyBatis-Plus 操作 MySQL，Flyway 管理表结构，Spring Security 接入登录态。我的后端能力重点在能把业务闭环跑通、能理解接口契约和数据模型，前端才是我更深入的部分。
```

## 后端内容怎么写更稳

可以写：

- 参与或负责 RESTful API 设计与联调。
- 能设计业务表、索引和唯一约束。
- 能用 Flyway 管理数据库迁移。
- 能实现登录、Token 校验、用户身份注入。
- 能处理第三方 API 调用失败的降级逻辑。

不建议写：

- 精通高并发架构。
- 负责微服务治理。
- 负责分布式事务。
- 设计千万级数据系统。
- 深度优化数据库性能。

## 45 秒项目介绍

```text
行迹旅图是一个前后端分离的旅游地图规划项目，我的定位是偏前端的全栈。前端主要做 React 地图工作台，接入百度地图，实现景点点位、路线绘制、行程池和完整行程时间轴；数据请求用 TanStack Query 管理。后端用 Spring Boot 提供 REST API，MyBatis-Plus 访问 MySQL，Flyway 做数据库迁移，Spring Security 接入 Bearer Token。这个项目能体现我对复杂前端交互、接口联调、数据建模和完整业务闭环的理解。
```
