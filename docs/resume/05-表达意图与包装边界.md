# 表达意图与包装边界

## 为什么这样写

这个项目最适合表达三件事：

1. **前端复杂业务能力**：地图工作台不是简单列表页，包含地图、筛选、详情、行程池、路线结果和时间轴联动。
2. **真实工程闭环**：项目有前端页面、后端接口、数据库迁移、认证授权、第三方地图 API 和用户数据沉淀。
3. **偏前端的全栈理解**：你可以把前端讲深，同时说明自己能理解后端接口、数据库和认证链路。

## 简历包装原则

### 要突出

- 地图 SDK 接入和可视化能力。
- TanStack Query 管理复杂服务端状态。
- React 组件拆分和页面状态编排。
- 前后端接口联调和统一响应处理。
- 登录态、收藏、打卡、保存行程等完整业务链路。
- MySQL 表设计、Flyway 迁移、Spring Security 基础认证。

### 要克制

- 不把项目包装成已大规模上线产品。
- 不写没有数据支撑的性能指标。
- 不把百度地图路线规划说成自研路线算法。
- 不把后端能力写成资深后端架构能力。
- 不写项目不存在的 Redis、消息队列、微服务、分布式事务。

## STAR 和数据怎么用

可以用，但这个项目更适合“弱 STAR + 真实结果”写法，不适合硬凑商业指标。

### 适合怎么写

- Situation / Task：一句话交代场景，例如“地图工作台需要把城市、景点、路线和行程展示串成闭环”。
- Action：重点写你怎么做，例如“封装百度地图 SDK、拆分业务组件、用 TanStack Query 管理服务端状态、补齐登录态与分享链路”。
- Result：写真实可验证的结果，例如“完成从选城市到保存/分享行程的闭环”“把路线与完整行程统一映射为可回放的前端展示模型”“让分页和不分页列表共享同一套前端消费结构”。

### 不适合怎么写

- 不要硬写“提升转化率 20%”“减少接口耗时 50%”这类没有日志、监控或实验依据的数据。
- 不要把“个人做完一个功能”硬包装成“大规模线上业务增长”。
- 不要为了符合 STAR 模板，把每条项目经历都写成很重的背景铺垫；前端简历更重要的是技术动作和结果是否扎实。

### 可以使用的结果表达

- 完成“城市浏览-景点筛选-路线规划-行程保存/分享”的功能闭环。
- 将地图工作台拆分为多个业务组件，降低超大页面的状态耦合。
- 统一前后端分页响应结构，减少页面层对接口差异的额外判断。
- 为路线规划增加第三方 API 失败时的本地估算降级，保证主流程可用。

### 如果确实要写数据

只有满足下面条件时再写：

- 数据来自代码、测试、构建产物或你能现场解释清楚的统计口径。
- 数据是工程结果，不是业务瞎估。

这类数据可以写：

- “核心页面拆分为 10 个业务组件和 3 个 Hook 模块”
- “后端维护 13 个 Flyway 迁移脚本管理表结构演进”
- “管理端用户、城市、景点列表统一支持分页与筛选”

## 前端岗位表达意图

前端岗位最应该让面试官看到：

```text
这个候选人不是只会写静态页面，也不是只会调接口。他处理过地图 SDK、复杂组件联动、服务端状态管理、登录态功能和前后端联调。
```

所以前端版简历要把篇幅放在：

- 百度地图接入。
- 地图工作台联动。
- TanStack Query。
- 路线和完整行程展示。
- 组件拆分和工程结构。

后端只作为“了解接口和业务闭环”的补充，不抢主线。

## 全栈偏前端表达意图

全栈偏前端岗位要表达：

```text
我以前端为主要优势，但不是只会等后端接口；我能理解业务数据怎么落库、接口怎么设计、认证怎么打通，并能把一个完整功能从页面做到接口和数据。
```

可以重点讲：

- 前端主导地图工作台。
- 后端提供业务 API。
- 数据库支持用户行程、收藏、打卡和路线持久化。
- Token 认证贯通前后端。
- 第三方地图 API 失败时有兜底。

## 面试讲述顺序

推荐按这个顺序讲项目：

```text
1. 项目背景：解决用户在一个城市里发现景点、规划路线和沉淀足迹的问题。
2. 核心功能：城市景点浏览、地图点位、路线规划、完整行程、收藏打卡、我的行程和公开分享。
3. 我的职责：以前端地图工作台为主，同时参与后端接口、数据库和认证联调。
4. 技术难点：地图 SDK 接入、坐标转换、路线绘制、多组件联动、Query 状态管理。
5. 工程设计：前端分层、请求封装、类型定义、后端 REST API、Flyway 迁移。
6. 反思优化：继续拆分超大页面、补充测试、优化地图重绘和移动端体验。
```

## 可放心讲的内容

- React + TypeScript + Vite 前端工程。
- Ant Design + CSS Modules 页面实现。
- TanStack Query 查询、变更、条件请求和缓存。
- 百度地图 SDK 动态加载、坐标转换、Marker、Polyline。
- 路线分段、Day 时间轴和行程保存。
- Spring Boot REST API。
- MyBatis-Plus + MySQL。
- Flyway 迁移脚本。
- Spring Security + Bearer Token。
- 收藏、打卡、公开分享和后台管理。

## 不要主动讲太满的内容

- 高并发、高可用、分布式架构。
- 自研路径最短算法。
- 线上大规模用户数据。
- 完整商业运营闭环。
- AI 个性化攻略。
- Redis 缓存优化。
- 微服务治理。

如果面试官问到，可以这样回答：

```text
这些属于后续可以扩展的方向。当前项目重点是把地图浏览、路线规划、用户行程和前后端数据闭环做完整，我没有把它包装成高并发或微服务项目。
```

## 针对 JD 的微调建议

### JD 强调 React

多写：

- 组件拆分。
- Hooks 状态组织。
- TypeScript Props 和 DTO。
- 复杂页面联动。

少写：

- 后端表结构细节。
- Spring Security 实现细节。

### JD 强调地图可视化

多写：

- 百度地图 SDK。
- 坐标转换。
- Marker/Polyline。
- 路线端点和方向标识。
- 地图选点。

### JD 强调全栈

多写：

- RESTful API。
- MySQL 数据模型。
- Flyway。
- Token 认证。
- 第三方 API 失败兜底。

但仍要保留“偏前端”的定位，避免面试官按纯后端深度追问。

## 最终推荐投递话术

前端岗位：

```text
我最近做的项目是一个旅游地图规划平台，主要负责 React 前端。项目里最核心的是地图工作台，涉及百度地图 SDK 接入、路线绘制、景点筛选、行程池和完整行程时间轴。我也做了前后端接口联调，所以对后端接口、Token 和数据模型有一定了解。
```

全栈偏前端岗位：

```text
这个项目我以前端为主，同时把后端接口和数据库也跑通了。前端负责地图交互和路线可视化，后端用 Spring Boot 提供城市景点、路线规划、用户行程和收藏打卡接口，MySQL + Flyway 管理数据。我的优势是能从前端体验出发，把接口和数据闭环一起考虑。
```
