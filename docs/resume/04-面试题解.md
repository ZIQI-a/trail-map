# 面试题解

## 前端高频题

### 1. 这个项目里你主要负责什么？

面试官考点：判断你是否真的做过核心模块。

推荐回答：

```text
我主要负责前端地图工作台和前后端联调。地图工作台包括城市切换、景点列表、标签筛选、地图点位、景点详情、行程池、路线规划结果和完整行程时间轴。除此之外，我也接入了收藏、打卡、我的行程和公开分享等登录态功能。技术上主要用了 React、TypeScript、TanStack Query、Ant Design 和百度地图 JS API。
```

项目结合点：`MapWorkbenchPage` 组织页面状态，`BaiduMapStage` 负责地图，`useMapWorkbenchData` 负责请求 hooks。

可追问点：为什么没有把所有状态放 Zustand？回答：当前大部分状态是页面局部状态，服务端状态交给 TanStack Query；只有跨页面或全局共享明显时才需要全局 store。

### 2. 地图 SDK 是怎么接入的？

面试官考点：是否只会复制示例代码，还是做了工程化封装。

推荐回答：

```text
我没有在组件里直接写 script，而是封装了 loadBaiduMapGL。它会判断 window.BMapGL 是否已存在，避免重复加载；如果没有 AK 会直接 reject；加载过程用一个全局 promise 保证整个应用只注入一次脚本。地图组件只依赖这个加载函数，初始化后再做城市中心定位、Marker、路线折线和选点监听。
```

项目结合点：`frontend/src/lib/baiduMap.ts`。

可追问点：为什么要单例加载？回答：地图 SDK 是全局资源，重复注入会增加不确定性，也可能导致回调覆盖和重复初始化。

### 3. 坐标系怎么处理？

面试官考点：地图业务是否真实。

推荐回答：

```text
项目内部坐标以 GCJ-02 为主，但百度地图底图使用 BD-09，所以创建百度地图点位前会做 GCJ-02 到 BD-09 的转换。地图点击选点拿到百度坐标后，再转回 GCJ-02 存入业务状态，保证前后端和数据库里的坐标体系一致。
```

项目结合点：`utils/map-workbench/coordinate.ts`、`createBaiduPoint`、地图选点逻辑。

可追问点：为什么不能直接混用？回答：混用会导致点位偏移，路线和景点位置对不上。

### 4. 路线是怎么画到地图上的？

面试官考点：复杂可视化数据如何落地。

推荐回答：

```text
后端路线规划接口返回分段路线，每段包含起点、终点、距离、耗时、路线说明和 polyline。前端把这些分段转换成地图 overlay，用百度地图 Polyline 绘制。为了提高可读性，还加了路线端点、方向箭头和多路线轻微偏移，完整行程模式下会按当前 Day 只展示当天路线。
```

项目结合点：`BaiduMapStage`、`buildMapRouteOverlays`、`resolveScheduleMapSegments`。

可追问点：为什么要按 Day 展示？回答：完整行程路线多时全部画出来会干扰阅读，按 Day 展示能和右侧时间轴联动。

### 5. 为什么用 TanStack Query？

面试官考点：是否理解服务端状态。

推荐回答：

```text
这个项目有大量服务端数据，比如城市、景点、标签、景点详情、POI 候选、收藏状态、打卡状态、行程详情和公开分享。如果都用 useEffect 手写请求，缓存、加载态、错误态和重复请求会很难维护。TanStack Query 可以通过 queryKey 表达数据依赖，通过 enabled 控制是否触发请求，通过 placeholderData 保留上一份列表，减少页面切换时的闪烁。
```

项目结合点：`useMapWorkbenchData.ts`。

可追问点：mutation 后怎么更新？回答：项目里登录后会 invalidate 当前用户查询，收藏/打卡等操作可以通过 query invalidation 或局部状态刷新保持一致。

### 6. 复杂页面状态怎么拆？

面试官考点：组件设计能力。

推荐回答：

```text
我把状态分成三类：服务端状态交给 TanStack Query，比如城市列表、景点列表、行程详情；页面交互状态放在 MapWorkbenchPage，比如当前城市、选中景点、行程池、起点、路线结果；纯展示状态由子组件内部维护。这样页面层负责状态编排，业务组件通过 Props 接收数据和回调，避免组件直接互相依赖。
```

项目结合点：`MapWorkbenchPage` 页面入口和 `components/map-workbench` 组件目录。

可追问点：什么时候会用 Zustand？回答：当状态需要跨页面共享、多个远距离模块频繁读写，且不是服务端状态时再引入。

### 7. 登录态前端怎么处理？

面试官考点：认证链路是否完整。

推荐回答：

```text
登录或注册成功后，前端把 token 保存到 localStorage。通用 request 封装每次请求时读取 token，如果存在就加到 Authorization: Bearer token。地图工作台里收藏、打卡、保存行程等接口会根据是否登录控制请求和按钮行为，未登录时先弹出登录框，避免直接发 401 请求。
```

项目结合点：`lib/http.ts`、`lib/authToken.ts`、`AuthDialog`。

可追问点：token 放 localStorage 有什么风险？回答：主要是 XSS 风险，生产项目要加强输入输出安全和 CSP；也可以考虑 HttpOnly Cookie 方案。

## 全栈联调题

### 8. 前后端接口契约怎么保持一致？

推荐回答：

```text
后端统一返回 ApiResponse<T>，分页使用 PageResponse。前端 request 封装统一解析 success、message 和 data，业务层只拿 data。接口字段在 TypeScript 类型里维护，比如城市、景点、路线、行程等 DTO。后端接口变化时，同步更新类型和 api 封装，页面组件不直接拼复杂请求。
```

项目结合点：`wiki/06-API接口文档.md`、`lib/http.ts`、`types/mapWorkbench.ts`。

### 9. 如果后端接口报错，前端怎么展示？

推荐回答：

```text
request 会先解析后端统一响应，如果 HTTP 非 2xx 或 success 为 false，就抛出后端 message。页面层通过 Query 的 error 或 mutation catch 展示 Alert、message、notification 或空状态。对于非 JSON 错误页，也有兜底错误信息，避免页面直接崩掉。
```

项目结合点：`parseApiPayload` 和地图工作台的错误提示逻辑。

### 10. Vite 代理和环境变量怎么用？

推荐回答：

```text
前端通过 VITE_API_BASE_URL 控制接口基础地址，百度地图 AK 通过 VITE_BAIDU_MAP_AK 注入。这样开发、测试、部署可以用不同配置，避免把真实 key 写进源码。
```

项目结合点：`lib/http.ts`、`lib/baiduMap.ts`。

## 后端基础题

### 11. 后端整体分层是什么？

推荐回答：

```text
后端是 Spring Boot 分层结构：Controller 负责接收参数和返回统一响应，Service 负责业务逻辑，Mapper 负责数据库访问，Entity 映射数据库表，DTO 负责请求和响应结构。这样前端联调时接口边界比较清晰，业务逻辑也不会堆在 Controller 里。
```

项目结合点：`backend/src/main/java/com/trailmap`。

### 12. 数据库表是怎么设计的？

推荐回答：

```text
核心分几组：城市景点域有 city、spot、spot_tag、spot_tag_relation；路线规划域有 route_record、route_segment；用户域有 app_user、user_favorite_spot、user_checkin_spot、user_trip、user_trip_item。收藏、打卡用 user_id + spot_id 唯一约束防重复，行程节点支持 spot、lunch、rest、hotel 等类型，方便完整行程扩展。
```

项目结合点：`wiki/05-数据模型与数据库.md`。

### 13. 为什么用 Flyway？

推荐回答：

```text
项目表结构和种子数据会持续变化，如果手工改数据库，开发和测试环境容易不一致。Flyway 用版本化 SQL 管理迁移，新增字段或修正数据都通过新的 V 版本脚本完成，历史迁移不直接修改，便于追踪和复现环境。
```

项目结合点：`backend/src/main/resources/db/migration`。

### 14. Token 认证后端怎么接入？

推荐回答：

```text
后端有 BearerTokenAuthenticationFilter，从 Authorization 头里取 token，调用 AuthTokenService 校验，校验成功后加载用户信息，封装成 AuthUserPrincipal 写入 Spring Security 上下文。后续接口就可以根据认证信息和角色判断是否允许访问。
```

项目结合点：`BearerTokenAuthenticationFilter`、`SecurityConfig`。

### 15. 百度路线 API 失败怎么办？

推荐回答：

```text
路线规划服务优先调用百度地图服务端 API。如果没有配置服务端 AK，或者第三方接口失败，会回退到本地估算路线，用大圆距离和不同交通方式的保守速度估算距离和耗时。这样开发环境和异常场景下，主流程不会被完全阻断。
```

项目结合点：`RoutePlanServiceImpl`。

## 项目追问题

### 16. 这个项目最大的难点是什么？

推荐回答：

```text
最大难点是地图页面的多组件联动和数据状态管理。比如选中景点会影响地图 Marker、详情面板、收藏打卡状态；生成路线后又要影响地图路线、路线抽屉、完整行程时间轴和保存行程 payload。我的处理方式是把服务端状态交给 Query，把页面交互状态集中在页面入口，把纯展示逻辑拆到组件和 utils 里。
```

### 17. 如果继续优化，你会做什么？

推荐回答：

```text
前端上我会继续拆分 MapWorkbenchPage，把保存行程回放、公开分享回放、路线结果映射等逻辑下沉到 hooks 或 utils；地图路线绘制可以进一步做增量更新，减少每次 clearOverlays 后全量重绘。工程上可以补充更多组件测试和端到端测试。业务上可以增加路线排序策略、更多地图图层和更完善的移动端交互。
```

### 18. 这个项目和普通后台管理系统相比有什么不同？

推荐回答：

```text
普通后台系统更多是表格和表单，而这个项目的核心是地图交互和路线规划。前端需要处理地图 SDK、坐标系、路线折线、多组件联动、完整行程时间轴和登录态操作。它既有数据管理，也有比较强的空间可视化和用户交互复杂度。
```
